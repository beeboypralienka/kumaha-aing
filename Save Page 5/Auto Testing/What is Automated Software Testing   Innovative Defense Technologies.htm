What Is Automated Software Testing?

The term invokes many different meanings from members of the software development and testing community. To some the term may mean test-driven development and/or unit testing; to others it will mean using a capture/record/playback tool to automate their software testing. The term might mean custom developing test scripts using a scripting language such as Perl, Python, or Ruby, while for yet others Automated Software Testing might be related to performance/stress testing only or entirely focused on functional testing and/or security testing phases using various software testing tools or products.

Before getting into the details, it is important that the term is clearly defined so that “Automated Software Testing” takes on a unified meaning and is clearly understood. We define Automated Software Testing in the context of our work and experience, and its true meaning.

How IDT Defines Automated Software Testing

Generally all tests that are currently run as part of a manual testing program, whether functional, performance, concurrency, stress, or other, can be automated. It is possible to develop software that converts any type of manual software test existing today into an automated test. Whether the automation would yield the necessary return on investment (ROI) is another question that requires evaluation, and experience has shown that even though it is possible to automate all tests, not all tests are worth automating. Keeping ROI in mind, we recommend the following high-level definition of Automated Software Testing.

In a nutshell…

“Automated Software Testing is the application and implementation of software technology throughout the entire software testing lifecycle (STL), with the goal of improving STL efficiency and effectiveness.”

The Goal of Automated Software Testing

Automated Software Testing refers to automation efforts across the entire STL, with a focus on automating the integration and system testing efforts. The overall objective of Automated Software Testing is to design, develop, and deliver an automated test & re-test capability which increases testing efficiencies, and if implemented successfully, it can result in a substantial reduction in the cost, time, and resources associated with traditional test and evaluation methods and processes  or software intensive systems.

Benefits of Automated Software Testing

Automated Software Testing efforts can support all facets of the testing lifecycle, to include the various software testing phases, production support, and software standard assessment and verification. Automation can help expedite and streamline testing efforts. If implemented correctly, it can help save time, lower costs, and raise the quality of your software product.

An Adaptable Solution

Automated Software Testing, when effectively implemented, is not exclusively GUI capture/playback (as so many understand it to be) and is not limited to a specific phase of testing, nor is it specific to any particular vendor’s product or dependent on a particular architecture or software language for the software to be tested.

Diverse Application Support

We believe that Automated Software Testing, if implemented effectively, should meet the following criteria by supporting applications:

  • running on multiple computers
  • developed in different programming languages
  • running on different types of OS’s or platforms
  • which have GUI and those which do not (for example, message interface testing)
  • independent of network protocols such as TCP/IP, DDS, etc.

Automated Software Testing should follow the following automation framework recipes:

  • Support integration of multiple commercial testing tools, whether open source, in-house developed, or from different vendors (allowing new or better products to be utilized as they emerge in the market)
  • Support testing w/o having to install Automated Software Testing on the same computers as the application under test and be able to be distributed across computers
  • Support evaluation of current manual testing processes and ability to determine improvement opportunities using Automated Software Testing
  • Use and integrate existing commercial or open-source software products without mandating the use of a specific product
  • Support both new applications under development and existing applications
  • Support the entire testing lifecycle w/o requiring application of Automated Software Testing to the entire testing lifecycle
  • Support distributed testing across multiple computers and various system engineering environments
  • Support reuse of Automated Software Testing frameworks and components among programs, as applicable.

Implementation of Automated Software Testing requires a mini-development lifecycle, including test requirements, automation framework design and construction, automated test implementation/verification, and execution. If done correctly Automated Software Testing should yield reusable and expandable frameworks and components. Best practices for software development apply and should be adhered to as well.

Adapted from: Dustin, Elfriede, Thom Garrett, and Bernie Gauf. “Implementing Automated Software Testing: How to save Time and Lower Costs While Raising Quality”. Upper Saddle River, NJ: Addison-Wesley, 2009. 51-53.
