[IT] Continuous Integration (CI)

Awalnya di kantor sering terdengar kata Continuous Integration. Apa itu? saya juga tidak tahu mahluk apa itu. Setelah membaca beberapa bab dari buku Continuous Integration Improving Software Quality and Reducing Risk (forewords by Martin Fowler & Paus Julius), ada sedikit catatan yang ingin saya tulis untuk dibaca-baca di kemudian hari.

Continuous Integration (CI) merupakan suatu praktek pengembangan perangkat lunak, digunakan untuk aktivitas tim yang dilakukan berkali-kali. Bagaimana bentuk prakteknya? Mari kita lihat lebih lanjut.

 CI merupakan metodologi untuk pengembangan perangkat lunak, di mana anggota tim diwajibkan untuk sesering mungkin mengintegrasikan kerjaannya dengan developer yang lain. Biasanya ritmenya per hari atau sering disebut daily build. Setiap proses integrasi tersebut akan dilakukan pengujian secara otomatis untuk mendapatkan defect atau problem secepat mungkin dan selanjutnya diperbaiki. Banyak anggota tim menemukan bahwa pendekatan ini menyebabkan masalah integrasi berkurang secara signifikan dan memungkinkan anggota tim untuk mengembangkan perangkat lunak kohesif lebih cepat.

Dengan menggunakan metodologi pengembangan perangkat lunak (software development methodology) ini, maka para pengembang (developer) punya fase-fase yang lebih terstruktur sehingga proses manajerial dan kontrol dalam pembuatan perangkat lunak menjadi lebih baik.

CI membuat aktivitas yang berulang-ulang lebih mudah dan terintegrasi. Yang dimaksud dengan aktivitas disini adalah kompilasi, rebuild Database, eksekusi automated test dan inspeksi, deploy, dan pengiriman feedback ke user. Selain mengurangi proses manual yang berulang, CI juga dapat mengurangi resiko, membuat proyek lebih baik dan visibility, men-generate perangkat lunak yang dapat di-deploy kapan saja dan dimana saja.



Skenario dari CI sebagai berikut :
  1. Developer commit code ke version control repository. Kemudian CI melakukan polling ke repository untuk setiap perubahan yang terjadi.
  2. Setelah commit dilakukan, CI server mendeteksi perubahan pada version control repository. Kemudian CI server mengambil copy-an code terakhir dari repository dan mengeksekusi build script untuk diintegrasikan dengan perangkat lunak.
  3. CI server akan men-generate feedback, misalnya melalui email ke anggota proyek yang telah dispesifikasikan terlebih dahulu.
  4. CI server melanjutkan polling untuk setiap perubahan di version control repository.
  5. Contoh tools yang mengimplementasikan CI antara lain: Hudson, CruiseControl, dll
Ada 4 hal yang perlu diperhatikan dalam CI :
  1. Koneksi ke version control repository
  2. Build script
  3. Feedback mechanism (email, RSS, SMS)
  4. Proses untuk mengintegrasikan perubahan source code (manual/ CI server)


Kelebihan dan kekurangan CI
Kelebihan yang bisa didapatkan dengan menggunakan CI model sebagai berikut :
  1. Mengurangi proses manual yang berulang.
  2. Mengurangi resiko karena mendeteksi & memperbaiki masalah integrasi yang terus menerus.
  3. Membuat proyek lebih baik dan jelas.
  4. Menghasilkan perangkat lunak yang dapat di-deploy kapan saja dan dimana saja.
  5. Menghemat waktu dan biaya selama proyek berlangsung.
Namun, kelemahan CI model adalah :
  1. Memerlukan pengaturan awal terlebih dahulu tahap demi tahap.
  2. Memerlukan test kode untuk mencapai pengujian secara otomatis.
  3. Refactoring (melakukan perubahan pada kode program dari perangkat lunak dengan tujuan meningkatkan kualitas dari struktur program tersebut tanpa mengubah cara program tersebut bekerja) [5] dalam skala besar dapat menggangu karena dapat merubah basis kode.
  4. Membutuhkan biaya perangkat keras yang besar untuk membangun mesin yang mendukung.
