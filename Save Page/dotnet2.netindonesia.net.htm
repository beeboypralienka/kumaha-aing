Fatur,
 
tentang pergerakan itu sangat menarik sekali. Saya terus terang sempet 'pair programming' boongan lewat google docs sama Hendry untuk belajar lebih lanjut gimana ngerefactor Unit Test ke Unit Test bergaya BDD dengan AAA syntax. Intinya mengubah penggunaan fixture seperti yang aku jelasin di imel sebelumnya, sampe penggunaan mocking framework:
 
Ala NMock Expect(mockedDependencies.method(null)).IgnoreArguments().Return(bla) (arrange dan assert bergabung).
 Ke arah AAA syntax: mockedDependencies.Stub(a => a.method(null) ditemani dengan mockedDependencies.AssertWasCalled( a => a.method(null)) (memisahkan arrange dengan assert).
Lebih menarik lagi melihat kode yang bukan diimplementasi dengan test driven ke kode yang test driven (dependensi-nya diinjek dari luar, method yang pendek, class yang kecil-kecil, dsb). Mungkin kita bisa 'pairing' dan saling bertukar ilmu.
 
Kalo kata Roy Osherove di interviewnya sama Hanselman di Hanselminutes, unit test musti dilakukan di sebuah unit di mana unit-nya bisa sepenuhnya dikontrol oleh kita. Begitu ada third party library (external dll), atau database (stored proc?) atau infrastructure spesifik (file io), test tersebut bukan unit test lagi walaupun dilakukan dengan unit test tooling. Itu sudah menjadi integration testing.
 
CI (continuous INTEGRATION) akan menjalankan unit test (fast test) dan kita bisa memiliki stage kedua untuk integration testing (slow test). CI yang berjalan lama tidak akan efektif. Tapi pada saat  yang bersamaan kita tetep mau nge-detect masalah secepat mungkin untuk 'stop the line' (Kaizen: continuous IMPROVEMENT).

Ini yang aku belajar dari percakapan itu:
Kalo sekedar TDD kita biasanya membuat seperti ini
 
[TestFixture]
public class StudentTest
{
  [Test]
  public void EnrolBakalMasukinStudentKeClass()
  {
    //set semua dependensi sebagai mock
    ..
    //set expektasi di mock-nya sesuai dengan situasi (Context)
    ..
    //set system under testnya
    ..   
    // run
    ..
  }
 
  [Test]
  public void JoinBakalNgirimImelKeKepsek()
  {
    //set semua dependensi sebagai mock
    ..
    //set expektasi di mock-nya sesuai dengan situasi
    ..
    //set system under testnya
    ..   
    // run
    ..
  }
}
kalo bdd berubah jadi
 
[TestFixture]
public WhenEnrolStudent
{
  [SetUp]
  public void Context
  {
    //set semua dependensi sebagai mock
    ..
    //set return value di mock-nya sesuai dengan situasi
    ..
    //set system under testnya
    ..   
    // run
    ..
  }
 
  [Test]
  public void ShouldMasukinKeClass()
  {
    //assert apakah interaksinya bener di mock.
  }
}
 
 
Context menandakan situasi di mana sebuah kode akan di text. Perhatikan nama classnya pun menggambarkan situasi tersebut: "WhenEnrolStudent".
Struktur seperti ini dinamakan AAA: Arrange Act Assert
