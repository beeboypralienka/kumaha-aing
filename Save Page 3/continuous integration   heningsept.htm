Continuous Integration

Continuous Integration (CI) adalah praktik yang menjadi bagian dari Extreme Programming development process. Untuk yang sudah akrab dengan dunia software development, pasti tau bahwa biasanya tim software developer berkolaborasi menggunakan versioning system seperti Mercurial. Dalam kolaborasi ini, terdapat mainline (repositori pusat/integration server, tempat di mana semua code disatukan), dan ada working copy di komputer masing2 programmer.

 

Jika seorang programmer sudah selesai menyelesaikan bagiannya, sudah dites, dan ia ingin menyatukannya dengan mainline, bisa jadi para programmer lain pun sudah melakukan hal yang sama. sehingga programmer ini harus menarik perubahan code dari mainline untuk kemudian menyatukannya dengan pekerjaannya terlebih dahulu. Masalahnya, kalau ada konflik, dia harus memperbaikinya dulu sampai program bekerja dengan baik dan sinkron dengan yang ada di mainline. kalau sudah, baru dia bisa commit pekerjaannya dan mengupdatenya ke mainline. kalau code di mainline berhasil di-build dan berjalan baik di integration server, maka update dari si programmer tadi berhasil.

 

Kalau ada masalah/konflik, hal ini bisa diketahui di sisi programmer atau di integration server. Dalam continuous integration, error di kedua sisi ini harus cepat diketahui dan diselesaikan dalam waktu yang tidak lama.

“In a Continuous Integration environment you should never have a failed integration build stay failed for long. A good team should have many correct builds a day. Bad builds do occur from time to time, but should be quickly fixed.

The result of doing this is that there is a stable piece of software that works properly and contains few bugs. Everybody develops off that shared stable base and never gets so far away from that base that it takes very long to integrate back with it. Less time is spent trying to find bugs because they show up quickly.”

 

Some Key Practices of CI (taken from Martin Fowler)
1. Maintain a Single Source Repository
Catatan penting dalam repository, bukan hanya bersisi kumpulan file source code, tapi juga semua file yang berkaitan dengan project itu (database schema, install scripts, third party libraries, dll). sehingga ketika itu program mau dijalankan di komputer baru, tetap bisa jalan.

 

2. Automate the build
make sure you can build and launch your system using these scripts using a single command. Intinya, instead of melakukan langkah-langkah untuk build secara manual, kita bisa mendapatkan build dilakukan secara otomatis.

 

3. Make Your Build Self-Testing
Untuk memastikan build berjalan dengan baik, dalam automated build script dimasukan juga proses untuk testing secara otomatis. Sehingga, kalau ada error, bisa diketahui lebih awal.

 

4. Everyone Commits to The Mainline Every Day
Semakin sering seorang programmer commit dan push kerjaannya ke mainline, maka semakin cepat bisa diketahui kalau ada konflik dengan pekerjaan programmer lain. Selain itu, semakin mudah juga untuk mencari sumber konflik dibandingkan jika pengerjaan sudah banyak baru di-commit.

 

5. Every Commit Should Build The Mainline on an Integration Machine
Setiap commit, harus dipastikan build di mainline pada server pusat berjalan dengan baik. Kalau belum, maka programmer yang baru commit harus menyelesaikan masalah yang ada sampai build sukses.

“A continuous integration server acts as a monitor to the repository. Every time a commit against the repository finishes the server automatically checks out the sources onto the integration machine, initiates a build, and notifies the committer of the result of the build. The committer isn’t done until she gets the notification – usually an email.”

 

dan lain-lain. Sebenarnya masih ada beberapa poin lainnya, tapi silakan baca aja di artikel aslinya :D

 

Apa sih keuntungan menggunakan CI. Menurut Martin Fowler, keuntungan terbesar menggunakan CI adalah mengurangi resiko kegagalan. Memang CI tidak secara langsung menghilangkan error, tapi memudahkan untuk menemukan dan mengatasi error. Di integration server, build dan testing dilakukan berkala secara otomatis, jadi kalau ada error bisa langsung diketahui. Di sisi programmer, karena error cepat diketahui, maka untuk mencari letak error dia hanya perlu berurusan dengan sedikit code yang baru saja dikerjakan, jadi lebih mudah dilakukan.

 

Kalau mau menerapkan CI, ada beberapa tools yang bisa digunakan. Salah satunya adalah Jenkins, yang bisa diakses di http://jenkins-ci.org.
